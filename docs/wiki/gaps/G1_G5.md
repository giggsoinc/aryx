## Part A — Chunked Resolution (G1)

`resolution/chunked.py`, three streaming passes over a `ChunkBackend`
(Postgres: `store/chunk_backend.py`; in-memory fake in tests):

1. **Key pass** — stream records, persist `(block_key, record_id)` to
   `aryx_block_member` (10K batches). Skipped on resume when keys exist.
2. **Score pass** — per block: load ONLY that block's records
   (`WHERE id = ANY(...)`), score pairs, persist `aryx_match_edge`, mark
   `aryx_block_done` (the resume checkpoint). Memory high-water = largest
   block (≤ MAX_BLOCK 5000). Oversized blocks skip with WARNING.
3. **Cluster pass** — edges ≪ records: in-memory UnionFind over edges,
   stream one golden-record entity per component (G3 policy hook honored).

Legacy `resolve()` stays the fast path under `ARYX_ER_CHUNK_THRESHOLD`
(default 100K). Equivalence proven: identical cluster sets on the same
fixture; kill-after-10-blocks + resume = uninterrupted result.

## Part B — Stage Checkpoints (G5)

- **Migration 0021** — `aryx_run_stage(run_id, stage, status, ...)`.
- **`store/checkpoint_store.py`** — StageTracker; leftover `running` reads
  as `failed` on resume (the process died).
- **`pipeline/stages.py`** — StageRunner context guard; `orchestrate.py`
  `run_pipeline(resume_run_id=...)` skips done stages and reuses the run's
  landed data (no re-extract — landing was already idempotent per run).
- **`POST /admin/jobs/{job_id}/resume`** — returns the durable stage rows +
  resumable flag; caller re-submits with `resume_run_id`.
