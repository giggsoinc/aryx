## Problem

Zero entity-resolution test coverage. No published precision/recall numbers. Thresholds `_AUTO_MERGE=0.92`, `_ADJUDICATE_FROM=0.90`, `_REJECT=0.60` were never validated against labeled data.

## What Was Built

### Datasets (`benchmarks/datasets/`)
- **`fetch.py`** — Febrl 1-4 via `recordlinkage` package (no network, ships labeled pairs)
- **`make_mfg.py`** — 10K synthetic supplier records: legal suffix variants (Corp/Corporation/Inc/Incorporated/LLC/L.L.C.), word-order permutations, edit-distance 1-2 typos, 30% exact duplicates. Mirrors real sales demo domain.

### Harness (`benchmarks/run_bench.py`)
- Imports from `aryx.resolution.*` directly — never reimplements
- Metrics: pairwise P/R/F1, blocking recall (isolates G2 contribution), candidate pair count, wall time
- Two adjudication modes: `always_accept` / `always_reject` — bracket the LLM's possible contribution
- `--compare-legacy` flag: runs old single-prefix `block_key()` for before/after delta
- `--sweep` flag: grid over AUTO_MERGE × ADJUDICATE_FROM, emits CSV + markdown frontier table
- Appends self-describing rows to `docs/wiki/BENCHMARKS.md`

### Makefile targets
```
make er-bench        # full benchmark, appends to BENCHMARKS.md
make er-bench-quick  # febrl1 only, <60s, prints P=x.xx R=x.xx F1=x.xx
```

Quick mode output is parseable by the Conductor's regression gate (>1% drop = block merge).

### Funnel tests (`tests/test_resolution_funnel.py`)
13 tests total: 12 passing, 1 declared xfail.
- UnionFind: transitivity, singleton, empty, no-merge, path-compression
- Blocking recall: suffix variant pairing, transposition pairing, legacy vs multi-key delta
- End-to-end: 50-record fixture (10 clusters × 5), dissimilar rejection, single record, empty input
- `test_golden_record_order_independent` — **xfail (G3 open)**: current `golden_record()` is order-dependent when both records have competing non-null values. Flips green when survivor-smith lands.

**Headline G2 delta (Febrl1):** Δ recall = +7.6pp, blocking recall +12.4pp.

**LLM adjudication budget:** always_accept − always_reject = +5.6pp recall on Febrl1. That is the ceiling of what G10 can buy.

**Adjudication band analysis:** The [0.90, 0.92) band contains **8.7%** of true-positive mass on Febrl1 and **2.7%** on mfg-10k. Both are >2%, so no band widening is needed for G10. Hand to G10: sweep CSV at `benchmarks/sweep_febrl1.csv` (generate with `--sweep`).

**Scale note (mfg-10k):** precision drops from 0.98 (500 originals) to 0.84 (5000 originals) — at scale, distinct companies sharing adjective/industry vocabulary ("Allied Steel Corp" vs "Allied Steel Inc") collide above 0.92 string similarity. This is a scoring-stage finding, not a blocking finding: candidate count also grew 2× vs legacy (1.27M pairs, 93s wall). Feeds G7 (confidence model) — string similarity alone cannot separate near-name distinct entities; embeddings/attributes must contribute.

## FINDING — Blocking Recall Below Gate (G2 Reopen Request)

**Acceptance gate from spec:** blocking recall ≥ 0.95.
**Measured:** 0.852 (Febrl1), 0.856 (mfg-500).

This means **~14.5% of true match pairs never enter scoring** — they are permanently lost regardless of how good the scorer or adjudicator is.

Root cause candidates (for G2 to investigate):
1. Febrl records with `NaN` given_name or surname produce sparse/empty text → weak blocking keys
2. Phonetic variants where Soundex collision is needed but first token is a stopword or number
3. Multi-word company names where sorted token-set key is truncated at 32 chars

**Handoff to block-smith:** See `docs/wiki/handoffs/bench-master-g2-reopen-2026-06-11.md`.
This does NOT block G10 — log it as a P3 observation and proceed. G10 should note that its upper-bound recall ceiling is 0.852, not 1.0.
