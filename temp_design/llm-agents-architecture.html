The LLM agents — what they are, what they call, where their params come from

There is no monolithic "agent." Aryx has 13 narrow LLM call-sites + 2 embedding call-sites, each a small focused function with its own system prompt, schema, and tier. They all funnel through one Broker → one provider router → one wire path. This is the whole map, read straight from the code.

Live right now on EC2: provider = google menial = gemini-flash-latest answer = gemini-flash-latest endpoint = generativelanguage…/openai key set ✓ → every tier currently resolves to gemini-flash-latest over the OpenAI-compatible path.

1The stack — every call goes through these five layers

A call-site never talks to a model directly. It asks the Broker for a tier; the Broker picks a model; llm.py dispatches to the right provider; one HTTP function does the wire.

The "menial / answer" roles vs "frontier / cheap" tiers. The pipeline agents ask for a tier directly (frontier or cheap). The Ask + discovery agents go through llm_runtime.chat(role, …) where role is menial or answer — that picks the model name, but the underlying request is always tier cheap. So the two vocabularies meet at the model registration in _local_broker(): menial_model → cheap tier, answer_model → frontier tier.

3Where the params come from

"Params" = which model, which endpoint, which key, what budget. There are two broker-construction paths; both read the same live Settings state.

🛠 Pipeline path — _local_broker()

admin_api.py · used by ingest, extract, mapping, adjudicate, relationships

  1. Reads live config from llm_runtime.status() (provider, menial_model, answer_model, endpoint).
  2. Registers TWO ModelSpecs: menial_model → tier "cheap", answer_model → tier "frontier".
  3. Sets local=True only when provider is ollama; else attaches api_key_ref + endpoint.
  4. Embed config: nomic-embed-text on local Ollama (override via ARYX_EMBED_MODEL / _ENDPOINT).
  5. Secrets resolved by llm_runtime._RuntimeSecrets → the key typed in Settings.

💬 Ask path — _broker_for(model)

llm_runtime.py · used by ask term/synthesise + doc/table discovery

  1. chat(role) picks the model name: role=menial→menial_model, else answer_model.
  2. Builds a ONE-model Broker registered at tier cheap for that model.
  3. Same live _state dict — so a Settings change hits Ask and the pipeline instantly, no restart.
  4. Calls complete_text(..., "cheap", ..., think=False) — keeps fast models out of "thinking" mode.
  5. Every call logged to aryx_llm_call (role, model, provider, tokens, latency, error).
The source of truth for live params is the in-process _state dict in llm_runtime.py, seeded from env (ARYX_LLM_PROVIDER, ARYX_LLM_MENIAL_MODEL, ARYX_LLM_REASON_MODEL, ARYX_LLM_BASE_URL, ARYX_LLM_API_KEY) and overwritten live by the Settings panel via set_config(). The key is held in process memory only — never written to disk or git.

4The Governor — budgets & automatic downgrade

Every completion calls broker.charge(tier, tokens). Before each choose(), the Governor checks if the requested tier has blown its budget — if so it walks down the ladder to a cheaper tier.

frontier
budget 2,000,000 tok
→
mid
budget 10,000,000 tok
→
cheap
0 = unlimited
→
local
0 = unlimited
Budgets live in catalog.json. 0 or missing = unlimited. When a tier with a positive budget is exhausted, effective_tier() returns the next non-exhausted rung to the right. So a frontier blowout silently degrades to cheap rather than erroring — the job keeps running on a cheaper model.
Reality check on catalog.json today: its "models": [] is empty — the real models are registered at runtime by _local_broker() / _broker_for() from the live Settings state. So catalog.json currently only supplies the budgets and the embed config; the model roll-call comes from Settings. With provider=google + one model for both roles, frontier and cheap both resolve to gemini-flash-latest right now.

5Two end-to-end traces

📄 Ingesting a PDF

  1. tag (cheap) — label the fields.
  2. embed — vectorise chunks on local nomic-embed-text.
  3. extract_mentions (cheap) — one call per chunk, span-validated.
  4. mapping.categorize (frontier) — fit mentions to ontology types.
  5. adjudicate (frontier) — only on the ambiguous ER pairs.
  6. infer_relationship (frontier) — name edges between entities.
  7. All bounded by the 120s LLM + 300s per-doc timeouts.

💬 Answering a question

  1. _extract_terms (menial→cheap) — pull nouns/ids from the question + history.
  2. Deterministic graph retrieval in FalkorDB (no LLM).
  3. Always attach the workspace overview (kinds, counts, aim).
  4. _synthesise (answer) — write the grounded answer, cite entity ids.
  5. Strip <think>, log to aryx_llm_call, render markdown.
